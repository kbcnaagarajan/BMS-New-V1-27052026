1. SYSTEM OVERVIEW
================================================================================

KarthikAi-OMS is a multi-tenant Business Management System with role-based
access control (RBAC), supporting four primary user profiles:

  - Super Admin (platform-level)
  - Company Admin (tenant-level)
  - Project Manager (managed projects scope)
  - Employee / Staff (self-only scope)

The system manages:
  - Companies (tenants) with subscription/packaging
  - Client CRM (clients, contacts)
  - Project 360 (projects, tasks, milestones, deliverables, team)
  - Employee Operations (timesheets, leaves, attendance, expenses)
  - Collaboration (meetings, documents)
  - Finance (invoices, payments)
  - Issues & Risks (issues, risks, change requests, support tickets)
  - Client Portal (client-facing dashboard and project views)
  - Reports & Analytics
  - Administration (users, roles/permissions, company settings)

RBAC is implemented via Role model (M2M to User) with Permission model
(linked to Module). Data scoping uses centralized Q-object filters per
module per role type.

2.2 Authentication
-------------------
- Custom User model (email-based, no username)
- CustomUserManager with create_user / create_superuser
- USERNAME_FIELD = 'email'
- Login via email + password
- Session-based auth using Django's auth framework
- Login view: authenticate() -> auth_login() -> redirect('dashboard')
- Logout view: auth_logout() -> redirect('login')

2.3 Technology Stack
---------------------
- Backend: Django 5.2.14, SQLite
- Frontend: Bootstrap 5.3.2 CDN, Bootstrap Icons 1.11.3
- JS Libraries: jQuery 3.7.1, DataTables 1.13.7, Chart.js 4.4.1, Select2 4.1.0
- No Django forms used -- all form rendering is manual via template loops

3. USER PROFILES (4 TYPES)
================================================================================

3.1 Super Admin (user_type='super_admin')
-------------------------------------------
  Permission: Unrestricted access to entire system
  Data Scope: No filter (sees all companies, all data)
  Menus: Dashboard, all group menus, Companies, all CRUD modules
  Capabilities:
    - Create/Edit/Delete companies
    - Monitor all tenant activity
    - View all projects, tasks, invoices across all companies
    - Invite company admins via email token
    - Toggle company active/inactive
    - Change subscription status
    - View global reports
  Restrictions: None (full platform access)

3.2 Company Admin (user_type='company_admin')
-----------------------------------------------
  Permission: Full access within their own company
  Data Scope: company-scoped (all modules filtered by user.company)
  Menus: Dashboard, all group menus, all modules within their company
  Capabilities:
    - Full CRUD on all modules within their company
    - Manage users, roles, permissions within their company
    - View and edit company settings
    - Manage all projects, clients, contacts
    - View all employee data (timesheets, leaves, attendance, expenses)
  Restrictions:
    - Cannot access other companies' data
    - Cannot create companies
    - Cannot access Super Admin platform pages

3.3 Project Manager (user_type='project_manager')
---------------------------------------------------
  Permission: Manage assigned projects and related data
  Data Scope: Projects where project_manager=user, and all downstream data
             (tasks, milestones, deliverables, meetings, documents, team,
              issues, risks, change requests, support tickets, invoices,
              payments, expenses) scoped to those projects.
             Timesheets: company-scoped (can view all employee timesheets)
             Leaves/Attendance: self-only
             Users/Roles: company-scoped
             Contacts/Clients: company-scoped
  Menus: Dashboard, Clients, Contacts, Projects, Tasks, Milestones,
         Deliverables, Meetings, Documents, Team, Issues, Risks,
         Change Requests, Support Tickets, Timesheets, Leave, Attendance,
         Settings
  Capabilities:
    - View/edit assigned projects
    - Create/edit tasks within managed projects
    - View team members
    - Create/edit meetings, documents, milestones, deliverables
    - View all timesheets (company-scoped for TimesheetEntry)
    - Log own leave and attendance
  Restrictions:
    - Cannot manage projects they're not assigned as manager
    - Personal data (leaves, attendance) is self-only
    - Cannot delete most data (edit only)
    - Cannot access admin panels (users, roles)

3.4 Employee (user_type='employee' or 'staff')
-------------------------------------------------
  Permission: Self-only data + assigned project data
  Data Scope:
    Personal modules (timesheets, leaves, attendance, expenses): employee=user
    Tasks: assigned_to=user OR team_members__user=user
    Projects: team_members__user=user (projects they're team members of)
    Project-related (milestones, deliverables, meetings, documents, team,
      issues, risks, change requests, support tickets):
      project__team_members__user=user
    Users: pk=user.pk (self only)
    Contacts, Clients, Invoices, Payments, Roles, Settings: pk=None (no access)
  Menus: Dashboard, (scoped) Projects, Tasks, Timesheets, Leave,
         Attendance, Expenses, Meetings, Documents
  Capabilities:
    - Log timesheets (auto-assigned to self)
    - Apply for leave (auto-assigned to self)
    - Mark attendance (auto-assigned to self)
    - Log expenses (auto-assigned to self)
    - View assigned tasks and projects
    - View team members on their projects
    - View meetings and documents on their projects
  Restrictions:
    - Employee FK is always auto-set to self (readonly)
    - Cannot view other employees' data
    - Cannot access contacts, clients, invoices, payments, roles, settings
    - Cannot manage users or roles

5.1 Generic CRUD Engine
-------------------------
  Four unified functions in frontend_views.py:
    _module_list(request, key)    -> generic_module_list.html
    _module_form(request, key, pk)-> generic_module_form.html
    _module_detail(request, key, pk)-> generic_module_detail.html
    _module_delete(request, key, pk)-> generic_module_confirm_delete.html

  _MODEL_CONFIG dict maps each key to:
    model, list_fields, form_fields, redirect URL name

  Key benefits:
    - Automatic field introspection via _build_fields_meta()
    - FK dropdowns scoped via _get_company_fk_queryset()
    - Role-based scoping via _get_role_scope()
    - Auto-save for company, client, created_by, employee, invoice_number

6. RBAC SYSTEM
================================================================================

6.1 Models
-----------
  Module      - Screen/module for menu & permission control (key-based)
  Permission  - Granular permission: codename, permission_type, linked to Module
  Role        - Collection of permissions, linked to Company (null for system roles)
  User.roles  - M2M to Role

6.2 System Roles (seeded)
--------------------------
  10 system roles defined in seed_data.py SYSTEM_ROLES dict:
    1. super_admin       - __all__ modules, __all__ permissions
    2. company_admin     - __all__ modules, __all__ permissions (within company)
    3. management        - view/export on most modules (read-only oversight)
    4. delivery_manager  - view/export on most modules (read-only oversight)
    5. project_manager   - view/create/edit/export on project-related modules
    6. account_manager   - view/create/edit on client/finance modules
    7. employee          - view/create/edit on work-related modules
    8. hr_manager        - view/create/edit on employee/people modules
    9. finance_user      - view/create/edit/export on financial modules
    10. client_admin     - view only on portal modules
    11. client_user      - view only on portal modules

6.3 Permission Types
---------------------
  view, create, edit, delete, approve, export

6.4 User Methods
-----------------
  - has_module_access(module_key)        - bool
  - has_perm_in_module(key, perm_type)   - bool
  - get_module_permissions(module_key)   - list of permission type strings
  - get_accessible_modules()             - QuerySet of Module
  - is_company_admin_user()              - bool
  - is_super_admin (property)            - bool (is_superuser OR user_type)
  - is_project_manager (property)        - bool

6.5 Menu Building (rbac.py: build_user_menu())
------------------------------------------------
  - Queries all active menu-item modules
  - Separates standalone (Dashboard) from grouped items (key starts with 'group_')
  - For each group, checks children modules accessible to user
  - Groups with no visible children are excluded
  - Returns list of dicts with key, name, icon, path, permissions, children

6.6 SCOPE_CONFIG (rbac.py)
----------------------------
  Maps model labels (e.g. 'client_crm.Client') to filter paths
  (e.g. 'company', 'client__company', 'project__client__company')
  Used by get_company_scope_filter() for Q-object company scoping

7. DATA SCOPING RULES
================================================================================

7.1 Centralized Scoping (frontend_views.py: _get_role_scope())
----------------------------------------------------------------
  Single function that returns a Q() object filter per module key per user role.
  Called by all 4 generic CRUD helpers and all 18 custom views.

7.2 Scoping by Role
--------------------
  SUPER ADMIN:
    Q() - no filter, sees everything

  COMPANY ADMIN / MANAGEMENT / DELIVERY MANAGER / ACCOUNT MANAGER / HR / FINANCE:
    Company-scoped: {model_filter_path: user.company}
    Mapping in _company_scope_q():
      clients -> company
      contacts -> client__company
      projects -> client__company
      tasks/milestones/deliverables -> project__client__company
      meetings/documents/team -> project__client__company
      invoices -> client__company
      payments -> invoice__client__company
      expenses/attendance/timesheets/leaves -> employee__company
      issues/risks/change_requests/support_tickets -> project__client__company
      users/roles -> company
      settings -> pk (company.pk)

  PROJECT MANAGER:
    projects -> Q(project_manager=user)
    tasks, milestones, deliverables, meetings, documents, team, expenses,
      issues, risks, change_requests, support_tickets ->
      Q(project__project_manager=user)
    invoices -> Q(project__project_manager=user)
    payments -> Q(invoice__project__project_manager=user)
    timesheets -> Q(employee__company=company) [company-wide timesheet view]
    leaves, attendance -> Q(employee=user) [self-only]
    users -> Q(company=company)
    roles -> Q(company=company) | Q(is_system_role=True)
    settings -> Q(pk=company.pk)
    clients -> Q(company=company)
    contacts -> Q(client__company=company)

  EMPLOYEE:
    timesheets, leaves, attendance, expenses -> Q(employee=user)
    tasks -> Q(assigned_to=user) | Q(project__team_members__user=user)
    projects -> Q(team_members__user=user, team_members__is_active=True)
    milestones, deliverables, meetings, documents, team, issues, risks,
      change_requests, support_tickets ->
      Q(project__team_members__user=user)
    users -> Q(pk=user.pk)
    contacts, clients, invoices, payments, roles, settings -> Q(pk=None)

8. WORKFLOWS
================================================================================

8.1 Company Registration / Admin Invite
------------------------------------------
  1. Super Admin navigates to /companies/add/
  2. Fills company details, selects package (optional), enters admin email/name
  3. System creates Company record with trial subscription
  4. System generates 64-char random token and CompanyInvite record
  5. Token NOT emailed (requires email integration)
  6. Admin visits /invite/<token>/ - sets password, accepts invite
  7. System creates User (company_admin role), links as company admin
  8. Company activated (is_active=True, subscription_status='active')
  9. User auto-logged in, redirected to dashboard

8.2 Employee Timesheet Entry
-----------------------------
  1. Employee clicks "Add Timesheet" (timesheets/add/)
  2. Form auto-fills employee field to self (readonly)
  3. Employee fills week_start_date, week_end_date, notes
  4. System auto-sets employee = request.user for employee role
  5. On save, redirects to timesheet list
  6. NOTE: TimesheetEntry model queried in timesheet_list (custom view)
          but Timesheet model is used in generic CRUD form

8.3 Leave Application
----------------------
  1. Employee clicks "Apply Leave" (leaves/add/)
  2. Form auto-fills employee field to self (readonly)
  3. Employee selects leave_policy (dropdown -- requires seed data)
  4. Employee fills dates, reason, handover_to
  5. On save, redirects to leave list

8.4 Project Creation
---------------------
  1. User clicks "Add Project" (projects/add/)
  2. Selects client from company-scoped dropdown
  3. Selects project manager from user list
  4. Fills project details (name, code auto-generated, dates, budget, etc.)
  5. On save, redirects to project detail view
  6. Detail view shows tasks, team members, and project activity

8.5 Task Management
--------------------
  1. User can add task from /tasks/add/ or from project detail
  2. Assignee dropdown lists all active users
  3. Task has status workflow: todo -> in_progress -> done
  4. Task list with DataTable for sorting/searching
  5. Each project detail page includes scoped task table

8.6 Company Monitoring (Super Admin)
--------------------------------------
  1. Super Admin sees platform dashboard with global stats
  2. Can view all companies, their user/client/project counts
  3. Each company detail page shows projects, users, clients, invoices,
     support tickets
  4. Can toggle company active/inactive status
  5. Can edit company details and subscription

8.7 User & Role Management
----------------------------
  1. Company Admin navigates to User Management (/users/)
  2. Can add/edit/delete users within company
  3. Roles & Permissions (/roles/) managed per company
  4. System roles are shared (company=None), custom roles are company-scoped

8.8 Client Portal Workflow
------------------------------
  1. Client Admin/User logs in
  2. Sees portal dashboard with their projects and invoice summary
  3. Can view project list (/portal/projects/)
  4. View-only access (no edit/create/delete capabilities)

11. VALIDATIONS
================================================================================

11.1 Login
-----------
  - Email + password required
  - authenticate() validates credentials
  - Error message "Invalid email or password" on failure

11.2 Company Creation (Super Admin)
-------------------------------------
  - Only super admins can create (returns 403 otherwise)
  - Company name required
  - Admin email and first name required for invite creation
  - Password >= 6 chars on invite accept
  - Password confirmation must match

11.3 Generic Module Forms
---------------------------
  - @login_required on all views
  - pk lookup uses get_object_or_404 with scope filter (404 if not in scope)
  - Invalid module key returns HttpResponseForbidden
  - FK fields: set to None if blank
  - Boolean fields: checked via 'fn in data'
  - Numeric fields: default to 0/0.0 if blank
  - Date fields: set to None if blank
  - Text fields: default to '' if blank
  - Employee auto-set to self for employee/staff roles (overrides POST data)
  - Company auto-set on create for models with company field
  - Client auto-set on create for models with client field (first client in company)
  - created_by auto-set on create

11.4 Model-level Validations
------------------------------
  - Company: unique_together constraints on departments (company+name),
    designations (company+title)
  - CompanyInvite: unique_together on company+email
  - Module: unique key
  - Permission: unique_together on module+codename
  - Role: unique? None specified (duplicate names allowed per company)
  - User: unique email, unique employee_id (nullable)
  - Email normalization via CustomUserManager

11.5 URL Security
-------------------
  - All views decorated with @login_required (except login, logout, invite accept)
  - contacts_form was missing @login_required (FIXED in session)
  - company_create/company_edit/company_toggle_active check is_super_admin
  - company_detail checks is_super_admin or company_id match
  - settings_form checks company ownership
  - Data scoping prevents access to out-of-scope records
    (get_object_or_404 with scope filter = 404 on non-owned records)
